What is the orchestrator?

Orchestrator = the manager

It's main.py. It never fetches data or generates charts itself. Its only job is to coordinate — call the right subagent at the right time, pass data between them, and compile the final answer. Like a project manager who delegates all the real work.

What it does step by step
1Reads AGENTS.md — loads agent identity, rules, output format. This is the orchestrator's "memory".
2Reads subagents.yaml — knows it should use gpt-4o for itself, gpt-4o-mini for subagents.
3You type "tesla" → it calls GPT-4o with one prompt: "extract ticker symbol" → gets back TSLA.
4Calls researcher_run("TSLA") — this is a Python function call, not an LLM call. Researcher runs and returns data.
5Calls analyst_run(research_data, "TSLA") — passes the full research dict to analyst. Analyst runs and returns analysis + charts.
6Calls GPT-4o one final time — gives it research summary + analysis text → gets back a clean structured response.
7Prints the response, opens all chart PNGs automatically on Mac via subprocess.run(["open", chart]).
The orchestrator uses GPT-4o (expensive model) only twice — ticker extraction and final formatting. All the real work is done by cheaper gpt-4o-mini subagents.
What data exists and where it flows

Researcher → analyst

Returns a Python dict with three keys:

summary — LLM-written text description
csv — 180 rows of real OHLCV data
ticker — e.g. "TSLA"

Analyst → orchestrator

Returns a Python dict with two keys:

analysis — LLM buy/hold/sell recommendation
charts — list of 5 PNG file paths on disk

CSV → Docker sandbox

The CSV string is written to a temp file on disk, then mounted into the Docker container at /data/stock_data.csv. S&P 500 CSV is mounted at /data/sp500_data.csv. Chart code reads these files directly inside the container.

Docker sandbox → analyst

Container writes PNG files to /output/ inside the container, which is mounted to a temp directory on your Mac. After container exits, executor scans that directory and returns the file paths.

Key insight
The CSV with 180 rows of prices goes directly to the sandbox — the LLM never reads raw numbers. The LLM only reads the text summary the researcher wrote. LLMs are bad at arithmetic, good at reasoning. Raw numbers go to Python, text goes to GPT.
How researcher is started
1Orchestrator calls researcher_run("TSLA") — just a Python function call in the same process. No new process, no HTTP call, no message queue.
2Researcher reads stock-research/SKILL.md — this markdown file tells it exactly what to fetch and how to format output.
3Researcher calls yfinance.Ticker("TSLA").history(period="6mo") directly — stores raw DataFrame as CSV before any LLM involvement.
The LangChain agent loop inside researcher
4Sends system prompt (SKILL.md) + "research TSLA" to GPT-4o-mini with 4 tools bound: get_stock_price, get_stock_history, get_stock_news, get_stock_fundamentals
5GPT-4o-mini responds with tool calls. LangChain executes each tool, appends result to message history. Loop continues.
6Loop ends when GPT-4o-mini returns a message with NO tool calls — it has enough data and writes the summary text.
7Returns {"summary": "...", "csv": "Date,Open,High...", "ticker": "TSLA"} back to orchestrator.
Researcher closes when the LangChain loop ends — no explicit shutdown needed. It's just a function that returns a value.
How analyst is started
1Orchestrator calls analyst_run(research_data, "TSLA") — a Python function call. Analyst receives the full dict from researcher.
2Analyst immediately fetches S&P 500 data directly via yfinance.Ticker("^GSPC").history(period="6mo") for benchmark chart.
Two separate jobs — no LLM for charts

Job 1 — chart generation

Calls execute_code(chart_code, csv_data, sp500_csv_data) directly. Pre-written Python functions, no LLM involved. Generates all 5 charts: price+MA, RSI, MACD, volume, benchmark.

Job 2 — text analysis

Calls GPT-4o-mini with research summary + SKILL.md instructions. LLM reasons about fundamentals, technicals, news. Returns buy/hold/sell with confidence and reasoning.

Charts are hardcoded — not LLM-generated. This was a deliberate decision after seeing GPT hallucinate wrong indentation, wrong variable names, and wrong column names repeatedly. LLMs are bad at writing reliable pandas/matplotlib code.
How analyst closes
5After both jobs complete, returns {"analysis": "...", "charts": ["/tmp/.../TSLA_price_ma.png", ...]} to orchestrator. Function ends.
What is the sandbox and why does it exist?

Problem: running generated code is dangerous

If you run Python code directly on your machine, a bad script could delete files, read your API keys, crash the process, or run forever. The sandbox puts a hard wall around code execution using Docker.

How the container is started and stopped
1executor.py writes the chart code to a temp .py file and both CSVs to temp .csv files on your Mac.
2Calls docker.containers.run(image="stock-sandbox", detach=True, ...) — Docker spins up a fresh container from the pre-built image.
3Three volume mounts: script.py → /code/script.py, stock.csv → /data/stock_data.csv, sp500.csv → /data/sp500_data.csv, temp_dir → /output/
4Safety limits: network_disabled=True (no internet), mem_limit=512m (no memory bombs), timeout=30s (no infinite loops).
5Container runs python /code/script.py — generates 5 PNG files into /output/ which maps to your Mac's temp directory.
6Container exits when script finishes. executor.py calls container.remove() — container is destroyed completely.
7Executor scans temp output dir for .png files, returns their paths. Temp code and CSV files are deleted.
A brand new container is created for every single execution and destroyed immediately after. Nothing persists between runs. The only thing that survives is the PNG files in the temp output directory.
